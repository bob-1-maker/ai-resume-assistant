# 第一阶段前端边界收敛改造需求

## Feature Description

本次需求聚焦第一阶段架构收敛，不直接扩展新业务功能。
核心目标是收敛前端登录态、请求层、简历状态管理与首页编排组件职责，为后续 AI、导出、管理员后台扩展提供稳定基础。

本阶段重点治理以下问题：

- 前端存在重复请求封装与不统一的 API 访问方式
- 用户登录态读写分散在路由、页面、store 和本地存储中
- 简历 store 同时承担本地缓存、文件保存、数据库同步和 UI 状态，职责过重
- `ResumeBuilder.vue` 承担过多编排职责，后续扩展风险高

## Inputs And Outputs

### Inputs

- 当前 Vue 3 + Vite + TypeScript 前端代码
- 当前 Express + Turso/libSQL 后端接口
- 现有普通用户登录、简历编辑、模板切换、AI 调用、导入导出能力
- 现有 Jest 单元测试与 Playwright E2E 测试基线

### Outputs

- 统一的前端请求层
- 更清晰的登录态管理边界
- 收敛职责后的简历状态管理方案
- 拆分后的首页简历编辑编排结构
- 与本次改造对应的单元测试和 E2E 测试

## Page Flow And API Flow

### Page Flow

普通用户主流程：

1. 用户访问 `/login`
2. 登录成功后进入 `/`
3. 首页加载当前用户状态与简历数据
4. 用户在首页进行简历创建、加载、编辑、模板切换、预览与导出

管理员流程不作为本阶段主改造范围，但现有 `/admin-login -> /admin` 流程必须保持可用。

### API Flow

普通用户核心接口流：

1. 前端通过统一请求客户端访问 `/api/users/login`
2. 登录成功后统一保存并使用用户令牌
3. 前端通过统一请求客户端访问 `/api/users/me`
4. 前端通过统一请求客户端访问 `/api/resumes` 相关接口完成简历列表、详情、保存与删除

AI 相关接口流：

1. 前端仍通过统一请求客户端访问 `/api/ai/*`
2. 本阶段不重写 AI 业务逻辑，只要求前端接入路径与错误处理保持一致

## Exception Scenarios

- 浏览器环境、Vite 本地代理环境与 Tauri 环境下 API 基地址处理不一致
- `localStorage` 中 token 或用户信息为空、损坏或与当前会话不一致
- 简历本地缓存可读但远端数据访问失败
- Tauri 文件保存失败但不应阻断浏览器主流程
- 统一请求层落地过程中，旧请求封装与新请求封装并存导致鉴权或错误处理行为不一致
- `ResumeBuilder` 拆分后，模板切换、AI 面板、导出、模块排序或预览流程发生回归
- 第一阶段改造影响 AI、导出或管理员已有功能

## Acceptance Criteria

- 前端保留单一统一请求入口，移除重复或分散的请求封装
- 用户登录态读取、写入、清理路径统一，避免页面、路由和 store 多处分散管理
- 简历 store 职责收敛，避免同时承担过多 IO 与视图编排逻辑
- `ResumeBuilder.vue` 拆分为职责明确的结构，且现有主流程不回退
- 登录、简历核心主流程具备单元测试覆盖
- 登录、首页核心流程具备 E2E 回归覆盖
- AI、导出、管理员后台在本阶段不做大改，但必须保持现有可用性

## Scope Boundary

本阶段包含：

- 前端请求层统一
- 普通用户登录态管理边界收敛
- 简历 store 职责收敛
- 首页编排组件拆分
- 相应测试补齐与更新

本阶段不包含：

- AI 服务端能力重构
- 导出链路性能专项优化
- 管理员后台功能扩展
- 后端数据库结构重设计
- 新业务功能开发
